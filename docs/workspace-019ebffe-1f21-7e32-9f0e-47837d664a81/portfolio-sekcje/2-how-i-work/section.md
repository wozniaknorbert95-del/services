> META-README ecosystem, translated into 5 client-readable phases. This is process-as-proof:
> a named architect must show a repeatable method, not „freestyle freelancing".

## Copy (source of truth)

**Eyebrow:** // How I work
**H2:** A method, not a magic trick.
**Lead:** Every project runs through the same workflow my own systems use internally — the
same one that runs a live multi-agent business. You always know what's happening, what comes
next, and where your approval is required.

**5 phases:**

1. **Map** — *Automation Map (paid first step)*
   We find your two or three biggest time-and-money leaks, score the ROI, and agree the first
   system to build. You keep the document either way.
   → *Deliverable: Automation Map (PDF/board) + prioritised plan.*

2. **Architect** — *Design before code*
   I draw the target architecture: which agent does what, where data flows, where the human
   approves, what gets logged. Plain language + a diagram you could hand to anyone.
   → *Deliverable: architecture diagram + spec.*

3. **Build with an AI workforce** — *Agents do the heavy lifting, I hold the guardrails*
   The system is assembled by AI agents I orchestrate (planner → coder → tester → review),
   against fixed rules. That's why it's faster and leaner than billable hours.
   → *Deliverable: working system in a test environment.*

4. **Verify** — *Human-in-the-loop gates*
   Nothing goes live until it's tested on real scenarios and you've signed off. Approval gates
   are built into the system itself, not bolted on.
   → *Deliverable: tested system + sign-off.*

5. **Hand over & maintain** — *No lock-in*
   Every system ships with a README and a short walkthrough. Take it in-house, or keep me on a
   light monthly plan. Your choice — no dependency by design.
   → *Deliverable: README + handover + optional maintenance.*

**Note:** This is the same workflow (single source of truth, agent cards, fixed rules) that runs
my own production ecosystem — not a process invented for the brochure.
