## W2-B — AI with human approval, not autopilot

**Leak:** Automation risk / loss of control  
**Proof layer:** HITL, Trust & Safety, Agent OS / governance  
**Pillar:** P2 Trust / objection  
**UTM slug:** `human-approved-ai`

### Post body

```text
Small businesses do not need AI autopilot.
They need AI with brakes.

The fear is rational:
- What if it sends the wrong reply?
- What if it changes something without approval?
- What if nobody can explain what happened?

That is why every system I build follows one rule:
the system proposes, the human approves.

In practice, that means drafted replies wait for review, deployments wait for sign-off, and important actions are logged.

The goal is not to remove the owner from the business.
The goal is to remove repeated admin while keeping decisions under control.

Automation should make the business calmer, not more fragile.

If you want speed but do not trust black-box AI, start by mapping where approval gates must exist.

#B2B #automation #buildinpublic
```

## W4-B — What a good Automation Map contains

**Leak:** Not knowing what to automate first  
**Proof layer:** Automation Map method  
**Pillar:** P4 Process  
**UTM slug:** `good-automation-map`

### Post body

```text
The first automation question should not be:
“What tool should we use?”

It should be:
“Where is the business leaking time or money?”

A good Automation Map answers five things:

1. Which process is costing the most?
2. Which steps repeat often enough to systemise?
3. Where must a human approve the outcome?
4. What is the smallest useful build?
5. What should not be automated yet?

That last question matters.
Bad automation makes a messy process faster.
Good automation makes the right process easier to run.

The output should be a written plan you can keep — not a vague promise that everything is possible.

If you do not know what to automate first, do not start with tools.
Start with the map.

#B2B #automation #buildinpublic
```
