# Marketing Strategy — Quietforge / services.flexgrafik.nl

**Version:** 2.1  
**Status:** Canonical — binding for all agents and copy sessions  
**Language:** English (all public-facing B2B assets on this domain)

---

## 1. Core question the site must answer

Every page, section, and CTA exists to support one buyer decision:

> **"Can this person design and deploy a system that improves my business?"**

The site must **never** optimise for:

> "Can they build a nice website / chatbot / automation script?"

If copy reads like a feature list, it fails strategy review.

---

## 2. Positioning (single label)

### Primary identity

**Conversion Systems Architect**

Norbert designs end-to-end **client acquisition and operations systems** — not isolated deliverables. Websites, AI assistants, wizards, and automations are **components of a system**, never the headline product.

### Supporting descriptor (one only — subheads, not hero)

**Lead qualification & automation systems**

Use in subheads and SEO where NL buyers search (*leadautomatisering*, *klantenwerving*). Do not add a second competing descriptor.

### Anti-positioning (active on hero and `/founder/`)

> Not a web designer. Not an AI chatbot builder. The person who designs the system that runs your client acquisition end-to-end.

Forbidden labels must be **actively rejected** on-page — not only listed in docs.

### Exception handling (public copy)

> If you only need a brochure website with no qualification or automation, this is not the right fit — and I will tell you before you spend money.

Reduces fear that the architect will refuse a simple ask; increases trust when the full system *is* the fit.

### AI in copy hierarchy

- **Hero H1:** Conversion Systems Architect — no "AI-powered" in headline (AI fatigue).
- **Sub-line:** AI assistants and automation logic appear as **system components**, not the product name.

### Forbidden positioning (do not use as primary label)

| Avoid | Why |
|-------|-----|
| Web designer / developer | Commodity, price-sensitive |
| AI builder / chatbot guy | Feature vendor, not architect |
| Freelancer / agency | No system credibility |
| "I do everything" | Dilutes trust; no ICP fit |

### One-sentence value proposition (template)

> Conversion systems that qualify leads, automate bookings, and reduce admin work — built with custom workflows, AI assistants, and automation logic, with you in the loop on every send.

Adapt per page; never drop the **system → outcome** arc.

## 3. Ideal Customer Profile (ICP)

| Attribute | Target |
|-----------|--------|
| Geography | Netherlands (NL), English-friendly SMB |
| Entity | ZZP, small BV, professional services |
| Pain | Too much admin, inbox chaos, leaky lead flow, no repeatable sales process |
| Budget signal | Willing to start with paid discovery (Automation Map €290+) and project work €1,200+ |
| Not for | Enterprise procurement, pure design-only briefs, sub-€199 impulse buyers |
| Decision style | Wants proof before call; respects process and boundaries |

**Tone for ICP:** Calm, precise, outcome-led. Hard numbers where honest (`12h/week`, `€800–1,500/mo` opportunity cost). No hype, no jargon stack.

**WhatsApp-friendly:** Dutch SMB buyers often prefer async chat-first discovery over cold calls — WhatsApp agent is a primary L3 path, not a sidebar widget.

### ICP objection map (address before CTA)

| ICP signal | Likely objection | Address in |
|------------|------------------|------------|
| Budget €290 discovery | "What do I get for that fee?" | Hero hint, `/pricing/`, `/book-discovery/` |
| Trust | "Is this real or agency theatre?" | Results, Behind the scenes, Field reports |
| Timing | "How long until something runs?" | How it works, case study timelines |
| Scope creep | "Will this balloon?" | Wizard scope-lock copy, Automation Map artefact |
| Website-only need | "I just need a site" | Exception handling (§2) + Web Upgrade path |

## 4. Message hierarchy (non-negotiable)

Every page and section follows **Problem → System → Effect**.

```
Problem     What hurts in their business today (recognisable pain)
    ↓
System      What was designed — architecture, workflow, human gates
    ↓
Effect      Measurable or process-proof outcome (honest claims only)
```

### Anti-pattern (reject in review)

```
Feature → Feature → Feature
```

Examples of failure: listing "chat, wizard, VCMS, automation" without tying each to a business outcome.

### PAS section template (every section)

```
// EYEBROW (category)
H2 — Problem statement
.qf-lead — Agitate (cost of inaction)
[content] — System (what was built)
.qf-hint or CTA — Effect (outcome / next step)
```

## 6. Hero section contract

Every homepage hero (and product heroes) MUST include five elements in order:

1. **Who** — Conversion Systems Architect (named: Norbert)
2. **For whom** — NL small business / professional services
3. **Result** — Qualified leads, less admin, systems that run with human approval
4. **Proof** — Live system line (e.g. `6 systems built • Custom VCMS • AI-integrated workflows`)
5. **CTA** — Primary: Book Automation Map / Strategy path; Secondary: Explore proof

Secondary link must point to **proof or process**, not a random anchor.

## 7. Case study schema (binding)

Each case study — list card and detail page — uses this structure:

| Section | Content |
|---------|---------|
| BEFORE | Client situation, pain, constraints |
| PROBLEM | Specific bottleneck |
| SOLUTION | System architecture (diagram or prose) |
| BUILD | What was implemented (scope, duration) |
| RESULT | Outcome — process-proof, estimate, or delivered metric |
| STACK | Technologies (secondary, not hero) |
| LESSONS | What you'd do again / constraint learned |
| VIDEO | 60–120s walkthrough when available |

**Portfolio card fields (list view):** Problem · Scope · Duration · Architecture · AI · Automation · Result

No client names unless approved. Anonymise by default.

### Self-as-client framing (owner ecosystem)

When the only production deployment is Norbert's own stack, use transparent framing:

> Deployed in production in my own ecosystem (2+ years running). Same architecture I deploy for clients — not theory.

Stronger than fake client logos. Never imply a third-party client engagement that did not happen.

## 9. Proof & credibility standards

### Required proof types (prioritised)

1. Screen recordings / Loom (wizard, VCMS, client journey)
2. Process artefacts (Automation Map sample, safety playbook)
3. Architecture diagrams (ecosystem flow)
4. Quantified build metrics (wizard steps, fields, workflows — **verifiable only**)
5. Case studies with honest measurement lines

### Credibility metrics (examples — update when verified)

Each metric MUST include an **outcome translation** for SMB readers:

| Feature metric | Outcome translation (example) |
|----------------|-------------------------------|
| 14-step wizard | Qualifies a lead in under 2 min vs 3-day email ping-pong |
| 37 dynamic fields | Full context captured — discovery is 20 min not 60 |
| 8 workflow automations | ~12h/week less admin (estimate — label if unverified) |
| Custom VCMS | One command centre instead of five disconnected tools |
| AI support layer | Repeat questions answered before they reach you |

Use Intent Colors: time (`--fx-time`), money (`--fx-money`), order (`--fx-order`).

**Rule:** If a number cannot be verified in the codebase or delivery logs, mark as `estimate` or omit until confirmed.

### Video fallback (block 4)

When Loom/video is not yet recorded:

- 16:9 frame with terminal mock poster
- Hover-to-play when asset exists
- Static captions describing the flow (deliberate placeholder, not broken embed)

### Field reports (instead of testimonials)

2–3 sentences of **self-report from production** — not client testimonials:

> Last Tuesday the wizard scored 14 leads in 6 hours; I closed 3 without a single email exchange.

Defensible, no fake testimonial risk.

### Imagery rules

| Type | Allowed |
|------|---------|
| People | Typographic only — no stock faces |
| System | Screenshots, diagrams, terminal mocks |
| Studio | Only if shot yourself |

### Forbidden

- Fake client logos or testimonials
- `[X]` placeholder metrics on live pages
- Stock photography
- Anonymous agency "we" in hero/founder (see voice-mix §10)

## 10. Voice & copy rules

| Do | Don't |
|----|-------|
| Outcome-first sentences | Feature dumps |
| Named architect voice | Anonymous agency "we" |
| Process-proof honesty | Guaranteed ROI |
| Human-in-the-loop as moat | "Fully autonomous AI" hype |
| PAS (Problem–Agitate–Solution) per section | Multiple competing messages per section |
| One psychological goal per section | Kitchen-sink sections |

### Voice-mix (product company + solo architect)

| Context | Voice |
|---------|-------|
| Hero, Founder, About | **I** — named architect |
| System architecture, Wizard, VCMS | **The system** — impersonal product |
| Process, delivery | **We** — you + client partnership |

**Tech jargon on this domain:** Allowed only when explaining **system architecture** to a business reader — always paired with a plain-language outcome. No LangGraph/repo dumps on SMB pages.
